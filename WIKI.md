# ServiceBusPractice — Wiki

## Overview
A small sample solution that demonstrates using Azure Service Bus from both an API and a background Worker service. The solution contains two main projects:

- `Order.Api` — ASP.NET Web API that publishes and inspects messages on Service Bus.
- `Order.Worker` — .NET Worker Service that processes messages from Service Bus.

## Configuration
Configuration is stored in `Order.Api/appsettings.json` (and similarly in `Order.Worker` if present). Important keys:

- `ServiceBus:ConnectionString` — connection string for your Service Bus namespace
- `ServiceBus:QueueName` — queue used by the API and worker (default: `orders`)
- `ServiceBus:TopicName` — topic used by the API (default: `orders-topic`)
- `ServiceBus:SubscriptionName` — subscription name (default: `orders-sub`)

Example `appsettings.json` snippet:

```
{
  "ServiceBus": {
    "ConnectionString": "Endpoint=sb://...;SharedAccessKeyName=...;SharedAccessKey=...",
    "QueueName": "orders",
    "TopicName": "orders-topic",
    "SubscriptionName": "orders-sub"
  }
}
```

Do not commit real connection strings to source control.

## Run locally

1. Ensure the `ServiceBus:ConnectionString` is set in `Order.Api/appsettings.json` (or environment variables).
2. Start the API project (`Order.Api`) from Visual Studio or via `dotnet run` in its folder.
3. Start the Worker project (`Order.Worker`) if you want background processing.

## API Endpoints
Open `Order.Api/Controllers/OrdersController.cs` to see available endpoints. Typical operations:

- Publish an order message
- Inspect messages in queue or subscription (uses `ServiceBusInspector`)

Use tools like `curl`, Postman, or the Swagger UI (if enabled) to exercise the API.

## Inspecting messages
The `ServiceBusInspector` service provides methods to peek messages without removing them:

- `PeekQueueMessagesAsync(int maxMessages = 10)`
- `PeekSubscriptionMessagesAsync(string subscriptionName = "orders-sub", int maxMessages = 10)`

These are useful for debugging and verifying the message flow.

## Worker behavior
The worker (`Order.Worker`) reads messages from the configured queue/topic and processes them according to `Order.Worker/Services/Worker.cs`.

## Troubleshooting
- Verify network access to the Service Bus namespace.
- Ensure the connection string has appropriate rights (Send/Listen/Manage as needed).
- Check application logs in Visual Studio Output or the console for errors.
